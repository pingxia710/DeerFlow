# Lead Agent

<Callout type="info">
  NextOS is the AI-organization layer built on the DeerFlow runtime. Its Lead
  Agent keeps the user's goal, plan, progress, context, boundaries, and final
  judgment. The internal `command-room` profile delegates execution so this
  brain remains clear enough to direct large projects.
</Callout>

## Responsibilities

The Lead Agent receives user messages, maintains conversation state, decides
what work is needed, writes self-contained handoffs, reads returned AI results,
and gives the final response. Runtime is built on LangGraph/LangChain for state,
streaming, checkpoints, middleware, and tool calls; those mechanisms do not
replace model judgment.

The default and ordinary custom-agent profiles may use configured direct tools.
The `command-room` profile is intentionally different:

- it receives direct `file:read` tools (`ls`, `read_file`, `glob`, and `grep`)
  for current facts, but no file-write, shell, web, MCP, image-inspection, or
  skill-management execution tools;
- `task()` is always enabled, even if a client sends
  `subagent_enabled: false`;
- direct tools are limited to read-only investigation, delegation, necessary
  user clarification, and presenting completed files;
- the Lead Agent reads each complete result and chooses every next action inside
  the AI-owned planning and plan-execution flow.

Professional role context is developer-authored prompt text. It does not grant
DeerFlow tools. Each task is one ephemeral Codex CLI AI that owns its plan and
native tools, returns complete natural-language text, and ends.

## NextOS Command Room Flow

<Steps>

### Receive and orient

Record the human's interest, direction, non-goals, real-world permissions, and
return-to-human boundaries as the Goal Mandate. Keep current facts, boundaries,
and progress in the Chair's Operating Brief.

### Obtain one planning proposal

Give one `planner` the complete goal, facts, boundaries, permissions, and
observable completion criteria. It returns one coherent proposal before the
Chair dispatches execution or additional Planners. Any fact finding the
proposal needs can become its first directed execution phase.

### Run opposition

After the complete Planner proposal returns, send one `opposition` AI the
original brief and complete proposal for one challenge round. It looks for
hidden assumptions, counterevidence, failure modes, boundary misses, and
materially stronger alternatives without making the decision.

### Form and discuss the execution plan

Read the complete Planner and Opposition results, then synthesize one execution
plan in the Chair. Present it and pause for human discussion. Substantive
execution begins only after explicit natural-language confirmation; no program
field authorizes, blocks, or advances that conversation.

### Execute the confirmed plan in parallel

Give each execution AI a complete professional role, goal, relevant facts,
exact working, input, and output paths, authority, forbidden changes, observable
completion criteria, and requested natural result. Run independent workstreams
in parallel when useful. Six Command Room slots are capacity, not a task-count
target; do not overload one child with independently separable professional
domains just to keep the count low.

### Continue the current plan

Read task results and current facts directly, including files, code, logs,
plans, and artifacts. Results are information for the Chair to act on, not a
task-level acceptance event. Resolve ordinary implementation details and
continue the current plan without a required `verifier` role. When a phase-level
report changes what should happen next, ask one `project-manager` for a
next-stage proposal, give that complete proposal and the phase facts to one
`opposition` AI, then form and present the Chair's next-stage plan and pause for
human discussion. Also return to discussion if facts require a Goal Mandate,
direction, material-boundary, real-world-permission, or irreversible-consequence
change. The program does not enforce either conversation.

### Complete the plan

Read every complete natural result. End the work when the plan's actual
completion criteria are satisfied; otherwise dispatch the next useful bounded
task. Program metadata never makes that decision.

### Persist and respond

LangGraph persists messages and factual runtime state when configured. Return a
natural user-facing conclusion and present completed artifacts when needed.

</Steps>

## Persistent organization state

- **Goal Workspace** stores the complete Goal Mandate, Current Operating Brief,
  and the Chair-authored Current Organization Map as append-only records. The
  Map is a temporary organization judgment, not a program workflow.
- **Result Inbox** preserves every complete child return separately. Coalesced
  wake signals and explicit acknowledgements are delivery facts, not quality or
  completion judgments.
- **Workspace History** is an owner-scoped, on-demand, paginated read of those
  same append-only facts. It returns original bodies unchanged; it neither
  enters every Chair prompt automatically nor turns acknowledgement into
  acceptance, completion, deletion, or hiding.
- **Goal Cells** are recursive, owner-scoped child Threads with inherited
  mandates and local briefs. They may coordinate narrower Cells and return a
  complete result to their parent. Capability and workspace references do not
  grant permissions or imply shared writable storage.

## Model Selection

The Lead Agent model resolves in this order:

1. valid per-request `model_name` / `model`;
2. the active custom agent's configured model;
3. the first model in `config.yaml`.

Delegated task models are independent. Configure them under `subagents.model`
and `subagents.reasoning_effort`; they do not inherit the Lead Agent model.

```yaml
subagents:
  model: gpt-5.6-terra
  reasoning_effort: xhigh
  timeout_seconds: 3600
```

## Thinking And Plan Modes

`thinking_enabled` controls supported Lead Agent reasoning features. Ordinary
agents may enable `TodoMiddleware` with `is_plan_mode`. Command Room does not
use the program todo list as its workflow: it keeps progress in the lead AI and
delegates work through prompts.

## Custom Agents And Bootstrap

Ordinary custom agents can configure their name, instructions, skills, direct
tool groups, and default lead model. This is separate from
`subagents.custom_agents`, which defines professional context and an optional
Codex model for one-shot roles.

Bootstrap mode remains a minimal setup profile used to create ordinary custom
agents. It is not the Command Room task runtime.
