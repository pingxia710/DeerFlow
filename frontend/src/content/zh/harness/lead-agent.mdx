# Lead Agent

<Callout type="info">
  NextOS 是构建在 DeerFlow 运行时上的 AI 组织层。它的 Lead Agent 保留用户
  目标、方案、进度、上下文、边界和最终判断；内部 `command-room` profile
  把执行交给子 AI，让这个脑袋能持续指挥大型项目。
</Callout>

## 职责

Lead Agent 接收用户消息、维护会话状态、判断需要什么工作、编写完整交接
prompt、阅读各 AI 返回的自然结果，并给出最终回复。LangGraph/LangChain
负责状态、流式传输、检查点、中间件和工具调用，但这些机制不代替 AI 判断。

默认 Agent 和普通自定义 Agent 可以使用配置的直接工具；`command-room`
有意采用不同边界：

- 获得用于读取当前事实的 `file:read` 工具（`ls`、`read_file`、`glob`、`grep`），
  但不获得文件写入、Shell、网页、MCP、图像检查或 skill 管理等直接执行工具；
- 即使客户端发送 `subagent_enabled: false`，`task()` 也始终启用；
- 直接工具只保留只读调查、委派、必要的用户澄清和呈现已经完成的文件；
- Lead Agent 阅读每个完整结果，并在由 AI 掌握的规划与方案执行主干内决定每一个
  下一步。

专业角色是开发者写入 prompt 的上下文，不授予 DeerFlow 工具。每个任务只
启动一个临时 Codex CLI AI，由它自主规划和选择原生工具，返回完整自然语言
结果后结束。

## NextOS 指挥室流程

<Steps>

### 接收并定位

在主会话中保持用户目标、确认过的上下文、当前边界和进度。

### 取得规划方案

实质性工作先把完整目标、事实、边界、标准和返回要求交给 `planner`。需要探索
多条路线时，可以并行启动多个独立 Planner，但它们都属于规划视角。

### 运行反方

完整方案返回后，把原始 brief 和完整方案交给 `opposition` AI 跑一轮挑战。
它检查隐藏假设、反证、失败路径、边界遗漏和实质性更强的替代方案；没有足以
改变方案的问题时可以直接说明，不为反对而反对，也不代替 Chair 决策。合成
后若核心方向改变，再跑一轮反方；局部修订不重复。

### 合成并确认执行方案

Chair 阅读 Planner 和 Opposition 的完整结果，合成一份执行方案：目标、范围、
边界、关键决策、完成标准、风险和等待人决定的事项。与人讨论方案并等待自然语言
中的明确执行授权后，才向执行 AI 写全专业角色、目标、相关事实和路径、权限、
禁止事项、完成标准及返回要求。独立执行可以并行，由 Chair 决定有价值的
`task()` 数量；DeerFlow 不会用程序截断或延后调用。

### 继续已确认方案

直接阅读任务结果和当前事实，包括文件、代码、日志、方案和工件。结果是供 Chair
处理的信息，不是逐任务验收事件。在已确认方案内自行处理普通实现细节、继续推进，
不要求 `verifier` 角色。只有事实要求改变核心方向、范围、权限或业务决策时，
才回到人与方案的讨论。

### 完成方案

阅读每个 AI 返回的完整自然语言结果。只有方案的实际完成标准满足时结束工作；
否则继续派发下一个有价值且边界清楚的任务。程序元数据不做判断。

### 持久化并回复

配置检查点后，LangGraph 保存消息和客观运行状态。向用户返回自然结论，并
在需要时呈现已经完成的产出物。

</Steps>

## 模型选择

Lead Agent 模型按以下顺序解析：

1. 有效的请求级 `model_name` / `model`；
2. 当前自定义 Agent 配置的模型；
3. `config.yaml` 中第一个模型。

子任务模型独立配置，不继承 Lead Agent：

```yaml
subagents:
  model: gpt-5.6-terra
  reasoning_effort: xhigh
  timeout_seconds: 3600
```

## 思考与计划模式

`thinking_enabled` 控制 Lead Agent 支持的推理能力。普通 Agent 可以通过
`is_plan_mode` 启用 `TodoMiddleware`。指挥室不把程序 todo list 当作工作流；
进度由主 AI 保持，工作通过 prompt 委派。

## 自定义 Agent 与 Bootstrap

普通自定义 Agent 可以配置名称、指令、skills、直接工具组和默认 lead 模型。
它与 `subagents.custom_agents` 不同，后者只定义一次性专业角色的 prompt 上下文
和可选 Codex 模型。

Bootstrap 是创建普通自定义 Agent 时使用的最小设置模式，不是指挥室任务运行时。
