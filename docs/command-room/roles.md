# Command Room Professional Roles

Roles are developer-authored prompt context for short-lived intelligent agents,
not permanent processes or tool grants. The Chair selects and prompts every
role; containers describe the handoff purpose, not a fixed organization.

| Role angle | Typical handoff | Useful question |
| --- | --- | --- |
| planner | Optional Planning forward | What strong direction, goal, boundary, route, and acceptance shape follows from the Chair brief? |
| opposition | Optional Planning/Technical Design opposition | What contrary route, hidden assumption, boundary, or failure mode should the Chair see? |
| technical specialist | Optional Technical Design forward | What implementation route best serves the accepted goal? |
| executor | Execution cycle N | What bounded real work advances the goal, and what did it actually produce? |
| evidence / checker / reviewer | Review cycle N | What does independent inspection of the real result show relative to the goal? |
| recorder | Planning/Technical decision artifact | Which already-made Chair decision must be preserved unchanged? |
| stewardship/debt/freshness/learning/conflict/capability | Chosen by Chair | Which specialized project risk needs an independent view? |

Planning and Technical Design angles do not debate or review each other. Every
execution result receives a fresh review, but the Chair chooses review expertise
and depth according to the deliverable. Each role writes its assigned Markdown,
returns the same complete natural-language result, and ends.

Program code may enforce only explicit order, cycle identity, artifact receipt,
and hard permissions. It never treats a role result as approval or chooses the
next AI.
