# Lead Agent

<Callout type="info">
  NextOS 是构建在 DeerFlow 运行时上的 AI 组织层。它的 Lead Agent 保留用户
  目标、方案、进度、上下文、边界和最终判断；内部 `command-room` profile
  把执行交给子 AI，让这个脑袋能持续指挥大型项目。
</Callout>

## 职责

Lead Agent 接收用户消息、维护会话状态、判断需要什么工作、编写完整交接
prompt、阅读各 AI 返回的自然结果，并给出最终回复。LangGraph/LangChain
负责状态、流式传输、检查点、中间件和工具调用，但这些机制不代替 AI 判断。

默认 Agent 和普通自定义 Agent 可以使用配置的直接工具；`command-room`
有意采用不同边界：

- 获得用于读取当前事实的 `file:read` 工具（`ls`、`read_file`、`glob`、`grep`），
  但不获得文件写入、Shell、网页、MCP、图像检查或 skill 管理等直接执行工具；
- 即使客户端发送 `subagent_enabled: false`，`task()` 也始终启用；
- 直接工具只保留只读调查、委派、必要的用户澄清和呈现已经完成的文件；
- Lead Agent 阅读每个完整结果，并在由 AI 掌握的规划与方案执行主干内决定每一个
  下一步。

专业角色是开发者写入 prompt 的上下文，不授予 DeerFlow 工具。每个任务只
启动一个临时 Codex CLI AI，由它自主规划和选择原生工具，返回完整自然语言
结果后结束。

## NextOS 指挥室流程

<Steps>

### 接收并定位

把人的兴趣、方向、非目标、现实权限和返回讨论边界记录为 Goal Mandate，
在 Chair 的 Operating Brief 中保持当前事实、边界和进度。

### 取得一份完整规划方案

先把完整目标、事实、边界、权限和可观察完成标准交给一个 `planner`。它返回一份
连贯方案之前，Chair 不派发执行或额外 Planner。方案所需的信息收集可以成为它
规划出的第一阶段执行工作。

### 运行反方

Planner 的完整方案返回后，把原始 brief 和完整方案交给一个 `opposition` AI
跑一轮挑战。它检查隐藏假设、反证、失败路径、边界遗漏和更强替代方案，不代替
Chair 决策。

### 形成方案并与人讨论

Chair 阅读 Planner 和 Opposition 的完整结果，合成一份执行方案，呈现给人并
暂停讨论。只有自然语言明确确认后才开始实质执行；没有任何程序字段授权、阻止
或推进这段对话。

### 并行执行已确认方案

给每个执行 AI 写全专业角色、目标、相关事实、准确的工作地址、输入地址和输出
地址、权限、禁止事项、可观察完成标准及返回要求。有用的独立工作并行执行。
指挥室的 6 个槽位是资源容量，不是任务数量指标；不要为了少派任务，把可以独立
拆开的专业领域硬塞给一个子 AI。

### 继续当前方案

直接阅读任务结果和当前事实，包括文件、代码、日志、方案和工件。结果是供 Chair
处理的信息，不是逐任务验收事件。在当前方案内自行处理普通实现细节、继续推进，
不要求 `verifier` 角色。当阶段性报告改变下一步时，先让一个
`project-manager` 基于完整事实提出下一阶段方案，再把完整方案和阶段事实交给
一个 `opposition`，最后由 Chair 形成并呈现下一阶段执行方案，暂停与人讨论。
如果事实要求改变 Goal Mandate、方向、实质边界、现实权限或带来不可逆后果，
同样回到人与方案的讨论。这个顺序由 AI prompt 和自然结果实现，程序不强制执行。

### 完成方案

阅读每个 AI 返回的完整自然语言结果。只有方案的实际完成标准满足时结束工作；
否则继续派发下一个有价值且边界清楚的任务。程序元数据不做判断。

### 持久化并回复

配置检查点后，LangGraph 保存消息和客观运行状态。向用户返回自然结论，并
在需要时呈现已经完成的产出物。

</Steps>

## 持久化组织状态

- **Goal Workspace** 以追加式 AI 记录保存完整 Goal Mandate、Current
  Operating Brief 和 Chair 编写的 Current Organization Map；Map 是临时组织判断，
  不是程序工作流。
- **Result Inbox** 分别保存每个子 AI 的完整返回；合并唤醒信号和明确确认收悉
  只是交付事实，不是质量或完成判断。
- **Workspace History** 是对同一追加式事实的、归属 owner 的按需分页读取；它
  原样返回正文，不会自动进入每轮 Chair prompt，也不会把 acknowledgement 变成
  验收、完成、删除或隐藏。
- **Goal Cell** 是可递归、归属用户的独立子 Thread，继承 Mandate 并拥有本地
  Brief；它可以协调更窄的 Cell，再把完整结果返回父级。能力与工作区引用不授予
  权限，也不表示父子共享可写存储。

## 模型选择

Lead Agent 模型按以下顺序解析：

1. 有效的请求级 `model_name` / `model`；
2. 当前自定义 Agent 配置的模型；
3. `config.yaml` 中第一个模型。

子任务模型独立配置，不继承 Lead Agent：

```yaml
subagents:
  model: gpt-5.6-terra
  reasoning_effort: xhigh
  timeout_seconds: 3600
```

## 思考与计划模式

`thinking_enabled` 控制 Lead Agent 支持的推理能力。普通 Agent 可以通过
`is_plan_mode` 启用 `TodoMiddleware`。指挥室不把程序 todo list 当作工作流；
进度由主 AI 保持，工作通过 prompt 委派。

## 自定义 Agent 与 Bootstrap

普通自定义 Agent 可以配置名称、指令、skills、直接工具组和默认 lead 模型。
它与 `subagents.custom_agents` 不同，后者只定义一次性专业角色的 prompt 上下文
和可选 Codex 模型。

Bootstrap 是创建普通自定义 Agent 时使用的最小设置模式，不是指挥室任务运行时。
