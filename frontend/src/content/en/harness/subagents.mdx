# Subagents

<Callout type="info" emoji="👥">
  Subagents are focused workers that the Lead Agent delegates subtasks to. They
  run with isolated context, keeping the main conversation clean while handling
  parallel or specialized work.
</Callout>

NextOS is the AI organization built on the DeerFlow runtime. Its Lead Agent is the Command Room brain. It keeps the goal, plan, progress, context, and final judgment clear while one-shot **sub-AIs** execute work from self-contained natural-language prompts.

## Why subagents matter

Sub-AIs solve two key problems in long-horizon work:

1. **Context isolation**: a subagent only sees the information it needs for its piece of the task, not the entire parent conversation. This keeps each agent's working context focused and tractable.
2. **Parallelism**: multiple subagents can run concurrently, allowing independent parts of a task (e.g., researching multiple topics simultaneously) to be processed in parallel.

## Professional role labels

DeerFlow exposes `general-purpose`, `bash`, and reusable Command Room roles:
`planner`, `project-manager`, `executor`, `fact-finder`, `opposition`,
`recorder`, `runtime-reliability-auditor`,
`persistence-migration-auditor`, `frontend-protocol-auditor`,
`security-auditor`, and `platform-ops-auditor`. A role label selects
developer-authored context that DeerFlow prepends to the task: a configured
`system_prompt`, otherwise the role description. It does not grant DeerFlow
tools or create a turn loop; the one-shot Codex AI owns its native tools and
decisions.

### general-purpose

A general execution role for exploration, implementation, commands, checks, and other delegated work.

### bash

A command-execution role for builds, tests, diagnostics, and verbose terminal work. The label does not grant shell access; Codex CLI uses its own configured tools and permissions.

### planner

An independent planning role that turns the complete Chair brief into one
coherent proposal with goals, boundaries, assumptions, route, and observable
completion. It returns before execution or additional Planners begin.

### opposition

An independent challenge role used after a planner proposal. It receives the original brief and complete plan, then reports hidden assumptions, counterevidence, failure modes, boundary misses, and the strongest materially different alternative. It does not force disagreement, approve or reject the plan, or replace Chair judgment.

### project-manager

A one-shot phase-handoff role that proposes the next bounded stage from complete
phase facts. It never starts, approves, advances, or closes work. The Chair gives
its proposal to Opposition, forms the next-stage plan, and pauses for human
discussion.

### specialist auditors

The five fixed audit perspectives keep evidence methods narrow:
`runtime-reliability-auditor` covers Run/task/background/wake lifecycles;
`persistence-migration-auditor` covers database invariants and migrations;
`frontend-protocol-auditor` covers HTTP/SSE and browser state;
`security-auditor` covers trust boundaries; and `platform-ops-auditor` covers
deployment, readiness, observability, CI, and supply chain. They report evidence
and limits; they do not implement fixes or approve completion.

For substantive work, the Chair uses an AI-owned `planner` proposal → one
`opposition` challenge → Chair execution plan → human discussion and explicit
natural-language confirmation → plan execution flow. Returned task results are
facts for the Chair to use, not task-level acceptance events. This sequence
lives in AI prompts and natural results, not runtime state or frontend gates.

## Delegation flow

The Lead Agent delegates work to a subagent using the built-in `task` tool:

```
task(
  description="Research competitor pricing",
  prompt="""You are a B2B SaaS pricing researcher. Find Acme Corp's top five
competitors, verify their current pricing from primary sources, and return a
concise natural-language comparison with source links and any uncertainty.""",
  subagent_type="fact-finder"
)
```

The runtime then:

1. Assembles the developer-authored role context, the Lead Agent's task prompt, applicable instruction guidance, and exact working, input, and output paths, then passes that self-contained prompt to one ephemeral Codex CLI process.
2. Lets that Codex AI plan, use its tools, and complete the task autonomously.
3. Returns the complete natural-language result unchanged to the Lead Agent.
4. Ends the child process. The Lead Agent reads the result and decides whether any further AI work is useful.

## Goal Workspaces and recursive Goal Cells

Command Room persists a complete Goal Mandate, Current Operating Brief, and
Chair-authored Current Organization Map in an append-only Goal Workspace. Every successful child return remains a separate,
complete Result Inbox envelope. A wake may signal several envelopes together,
but reading or acknowledging them is only a Chair-authored fact, never approval
or completion.

When the root coordination context is genuinely insufficient, the Chair can
create a recursive Goal Cell. Each Cell is an owner-scoped child Thread that
inherits the current Goal Mandate, receives its own complete Operating Brief,
may create narrower Cells, and explicitly returns its complete result to its
parent. Capability and workspace references are structural facts only; they do
not grant permissions or create a shared writable workspace.

A `project-manager` is a one-shot planning perspective for a phase handoff: it
proposes the next bounded stage from complete results, but never starts,
approves, or advances work.

## Configuration

Codex task transport is configured through the `subagents:` section in `config.yaml`:

```yaml
subagents:
  # Lifecycle timeout for each Codex task (default: 3600 = 60 minutes)
  timeout_seconds: 3600
  # Explicit default model and reasoning effort for delegated tasks
  model: gpt-5.6-terra
  reasoning_effort: xhigh
  # Optional professional role names shown to the Lead Agent
  custom_agents:
    contract-inspector:
      description: Inspects API contracts and reports concrete mismatches.
      system_prompt: Act as an independent professional API contract inspector.
```

There is no DeerFlow turn loop, tool allowlist, score, or programmatic completion rule for child AIs. The assembled natural-language prompt is the task contract; the timeout only owns process lifecycle.

Delegation also requires the pinned `@openai/codex` CLI and `codex login`, or a supported Codex authentication environment. This remains true when the Lead Agent uses a different model provider. Docker images install the CLI but do not include personal credentials by default.

## Dispatch count

`subagent_enabled` controls whether delegation is active for a session. Once it
is active, the Lead Agent chooses useful `task()` calls from the confirmed plan
and current context. Independent plan work should run concurrently when useful,
but six Command Room slots are capacity, not a quota. Separable professional
domains should not be packed into one child merely to keep the count low.
Transport remains content blind: one Command Room may hold six queued or running
children, the Gateway runs twelve child processes, and up to sixty-four more
wait FIFO. These limits never parse prompts, rank work, choose roles, approve
results, or decide the next action; legacy `max_concurrent_subagents` request
values are ignored.

## Factual dispatch record

Each `task` call writes a compact append-only record under the thread's audit directory when local storage is available. The record keeps factual dispatch identity, status, hashes, and character counts while omitting raw prompts, worker output, and error text. It does not interpret worker prose or decide what the Lead Agent should do next.

When a child completes but the Chair wake fails, the UI can show that transport fact without treating it as Chair consumption, project completion, approval, or a retry request.

## ACP agents (external agents)

In addition to the built-in subagents, DeerFlow supports delegating to external agents through the **Agent Connect Protocol (ACP)**. ACP allows DeerFlow to invoke agents running as separate processes (including third-party CLI tools wrapped with an ACP adapter).

Configure ACP agents in `config.yaml`:

```yaml
acp_agents:
  claude_code:
    command: npx
    args: ["-y", "@zed-industries/claude-agent-acp"]
    description: Claude Code for implementation, refactoring, and debugging
    model: null
    # auto_approve_permissions: false
    # env:
    #   ANTHROPIC_API_KEY: $ANTHROPIC_API_KEY

  codex:
    command: npx
    args: ["-y", "@zed-industries/codex-acp"]
    description: Codex CLI for repository tasks and code generation
    model: null
```

Ordinary agent profiles may invoke ACP agents through the `invoke_acp_agent` built-in tool. The Command Room profile exposes only `task`, necessary clarification, and result presentation; it delegates equivalent execution through one-shot Codex tasks instead of calling ACP directly.

<Callout type="tip">
  ACP agents run as child processes managed by DeerFlow. They communicate over
  the ACP wire protocol. The standard CLI tools (like the plain `claude` or
  `codex` commands) are not ACP-compatible by default — use the adapter packages
  listed above or a compatible ACP wrapper.
</Callout>

## Custom professional roles

`subagents.custom_agents` can add a role name, description, developer-authored `system_prompt`, and optional Codex model override. Invoke it through `subagent_type="my-role"`; DeerFlow prepends that role context, then the Lead Agent's task-specific goal, context, boundaries, authority, definition of done, and requested result. Skills, tool groups, turn limits, and per-role timeouts do not control the one-shot Codex process.
