# 子 Agent

<Callout type="info" emoji="👥">
  子 Agent 是 Lead Agent
  委派子任务的专注执行者。它们以隔离的上下文运行，在处理并行或专业工作的同时保持主对话清晰。
</Callout>

NextOS 是构建在 DeerFlow 运行时上的 AI 组织。它的 Lead Agent 是指挥室脑袋，保留目标、方案、进度、上下文和最终判断；一次性**子 AI**根据自包含的自然语言 prompt 执行工作。

## 为什么子 Agent 很重要

子 AI 解决大型工作中的两个关键问题：

1. **上下文隔离**：子 Agent 只看到完成其任务所需的信息，而不是整个父对话。这保持了每个 Agent 的工作上下文专注且可控。
2. **并行性**：多个子 Agent 可以并发运行，允许任务的独立部分（例如同时研究多个话题）并行处理。

## 专业角色标签

DeerFlow 提供 `general-purpose`、`bash`，以及固定的指挥室角色：`planner`、
`project-manager`、`executor`、`fact-finder`、`opposition`、`recorder`、
`runtime-reliability-auditor`、`persistence-migration-auditor`、
`frontend-protocol-auditor`、`security-auditor` 和
`platform-ops-auditor`。角色标签会选择开发者写好的上下文，由 DeerFlow 放到
任务 prompt 前面：优先使用配置的 `system_prompt`，否则使用角色 description。
它不会授予 DeerFlow 工具或创建轮次循环；一次性 Codex AI 自己拥有并选择原生工具。

### general-purpose

通用执行角色，适合探索、实现、命令、检查和其他委派工作。

### bash

用于构建、测试、诊断和大量终端输出的命令执行角色。标签本身不授予 shell 权限；Codex CLI 使用自己的工具与权限配置。

### planner

独立规划角色，根据 Chair 的完整 brief 形成一个包含目标、边界、假设、路线和
可观察完成标准的连贯方案。它返回之后，才开始执行或增加其他 Planner。

### opposition

在 Planner 完整方案之后运行的独立挑战角色。它同时获得原始 brief 和完整方案，报告隐藏假设、反证、失败路径、边界遗漏和实质性最强的替代方案；不强行反对、不批准或否决方案，也不替代 Chair 判断。

### project-manager

阶段交接时的一次性角色，根据完整阶段事实提出下一阶段方案，但不启动、批准、
推进或关闭工作。Chair 把它的方案交给 Opposition，再形成下一阶段方案并暂停与人讨论。

### 专业审计角色

五个固定审计视角各自保持专业边界：`runtime-reliability-auditor` 检查
Run、任务、后台、唤醒生命周期；`persistence-migration-auditor` 检查数据库
不变量和迁移；`frontend-protocol-auditor` 检查 HTTP/SSE 和浏览器状态；
`security-auditor` 检查信任边界；`platform-ops-auditor` 检查部署、就绪、
可观测性、CI 和供应链。它们报告证据和边界，不实现修复，也不批准完成。

对于实质性工作，Chair 使用由 AI 掌握的“`planner` 提案 → 一轮 `opposition`
挑战 → Chair 执行方案 → 人讨论并用自然语言明确确认 → 执行方案”。子任务结果是
Chair 使用的事实，不是逐任务验收事件。这个顺序存在于 AI prompt 和自然结果中，
不进入运行时状态或前端门禁。

## 委派流程

Lead Agent 使用内置 `task` 工具将工作委派给子 Agent：

```
task(
  description="研究竞争对手定价",
  prompt="""你是 B2B SaaS 定价研究员。找出 Acme Corp 的前五个竞争对手，
从一手来源核实当前定价，返回简洁的自然语言对比、来源链接和仍存在的不确定项。""",
  subagent_type="fact-finder"
)
```

运行时然后：

1. 组合开发者写的角色上下文、Lead Agent 的任务 prompt、适用规则提示，以及准确的工作地址、输入地址和输出地址，再把这个自包含 prompt 交给一个临时 Codex CLI 进程。
2. 由这个 Codex AI 自主规划、调用工具并完成任务。
3. 把完整自然语言结果原样返回给 Lead Agent。
4. 结束子进程；Lead Agent 阅读完整结果，并自行判断是否还需要其他 AI 工作。

## Goal Workspace 与递归 Goal Cell

Command Room 在追加式 Goal Workspace 中保存完整 Goal Mandate、Current
Operating Brief 和 Chair 编写的 Current Organization Map。每个成功子任务的完整返回都作为一封独立 Result Inbox 信封
保留。一次唤醒可以同时通知多封信封，但读取或确认收悉只是 Chair 写入的事实，
不代表批准或完成。

当根协调上下文确实不足时，Chair 可以创建递归 Goal Cell。每个 Cell 都是归属
同一用户的独立子 Thread，继承当前 Goal Mandate，获得自己的完整 Operating
Brief，可以继续创建更窄的 Cell，并把完整结果明确返回父级。能力与工作区引用
目前只记录结构事实，不授予权限，也不创建父子共享可写工作区。

`project-manager` 是阶段交接时的一次性规划视角：它依据完整结果提出下一阶段，
但绝不启动、批准或推进工作。

## 配置

Codex 子任务的传输参数通过 `config.yaml` 中的 `subagents:` 部分配置：

```yaml
subagents:
  # 每个 Codex 子任务的生命周期超时（默认：3600 = 60 分钟）
  timeout_seconds: 3600
  # 所有委派任务的明确默认模型和推理强度
  model: gpt-5.6-terra
  reasoning_effort: xhigh
  # 可选：向 Lead Agent 展示的专业角色名称
  custom_agents:
    contract-inspector:
      description: 检查 API 契约并报告具体不一致。
      system_prompt: 作为独立的专业 API 契约检查者工作。
```

DeerFlow 不再为子 AI 设置轮次循环、工具白名单、评分或程序化完成规则。组合后的自然语言 prompt 就是任务契约；超时只负责进程生命周期。

委派还要求安装项目锁定版本的 `@openai/codex` CLI，并完成 `codex login` 或提供 Codex 支持的认证环境；即使 Lead Agent 使用其他模型 provider 也是如此。Docker 镜像会安装 CLI，但默认不包含个人凭证。

## 派发数量

`subagent_enabled` 控制会话是否启用委派。启用后，由 Lead Agent 根据已确认方案
和当前上下文决定有价值的 `task()` 数量。有用的独立方案工作应当并行，但一个
指挥室的 6 个槽位是容量，不是凑数指标；也不要为了少派任务，把可以独立拆开的
专业领域塞给一个子 AI。运输层只施加与内容无关的资源容量：一个 Command Room
最多保有 6 条等待或运行中的子任务，Gateway 同时运行 12 个子进程，另有最多
64 条按 FIFO 等待。这些限制绝不解析 prompt、排序工作、选择角色、批准结果或
决定下一步；旧请求中的 `max_concurrent_subagents` 值会被忽略。

## 事实调度记录

每次 `task` 调用会在本地存储可用时向该线程的 audit 目录写入一条追加式记录。记录只保留客观的调度身份、状态、哈希和字符数，不写入原始 prompt、worker 输出和错误文本，也不会解释 worker 文字或决定 Lead Agent 的下一步。

子任务完成而 Chair 唤醒失败时，UI 只会显示这一运输事实；它不表示 Chair 已消费、项目完成、批准或可以重试。

## ACP Agent（外部 Agent）

除内置子 Agent 外，DeerFlow 还通过 **Agent Connect Protocol (ACP)** 支持委派给外部 Agent。ACP 允许 DeerFlow 调用作为独立进程运行的 Agent（包括用 ACP 适配器包装的第三方 CLI 工具）。

在 `config.yaml` 中配置 ACP Agent：

```yaml
acp_agents:
  claude_code:
    command: npx
    args: ["-y", "@zed-industries/claude-agent-acp"]
    description: 用于实现、重构和调试的 Claude Code
    model: null

  codex:
    command: npx
    args: ["-y", "@zed-industries/codex-acp"]
    description: 用于仓库任务和代码生成的 Codex CLI
    model: null
```

普通 Agent profile 可以通过 `invoke_acp_agent` 内置工具调用 ACP Agent。Command Room profile 只暴露 `task`、必要澄清和结果展示工具；它把同类执行交给一次性 Codex 子任务，不直接调用 ACP。

<Callout type="tip">
  ACP Agent 作为 DeerFlow 管理的子进程运行，通过 ACP 协议通信。标准 CLI
  工具（如原始的 `claude` 或 `codex` 命令）默认不兼容
  ACP——请使用上面列出的适配器包或兼容的 ACP 封装器。
</Callout>

## 自定义专业角色

`subagents.custom_agents` 可以增加角色名称、说明、开发者写的 `system_prompt` 和可选 Codex 模型覆盖。通过 `subagent_type="my-role"` 使用该标签；DeerFlow 会先放入角色上下文，再接 Lead Agent 写的具体目标、上下文、边界、权限、完成标准和返回要求。skills、工具组、轮次限制和按角色超时不会控制一次性 Codex 进程。
