## 〇、大白话摘要(给非技术 Owner)

- 你只需做三件事:**说想要什么**(大白话)、**试用后说感觉对不对**、
  **批准不可逆的动作**(花钱、公开、删东西)。
- 其他一切——定目标、拆方案、派活、检查、验收——AI 干。
- AI 每轮开始前会告诉你这轮要干什么,结束后用大白话汇报发生了什么、
  你可以试什么、下一步是什么。你随时可以打断、插话、改主意,你说的
  每句原话都会被原样记住,AI 的所有目标都必须能指回你的原话。
- AI 答题、AI 判卷,但卷子挂在你能亲手试的东西上:能跑的程序、能打开
  的页面、能看的报告。不是 AI 自己说了算。

# 第一部分:设计观点(防漂移护栏)

> 这一部分是 2.0 的全部"为什么"。未来任何人(AI 或人)想改 2.0 的
> 任何结论,先在这里找到支撑它的观点;改结论必须同时说明哪条观点
> 失效了、证据是什么。

## 一、关于指挥的本质

**观点 1:纯指挥零接触 = 瞎指挥;全自己干 = 上下文淹没。**
两个极端 1.0 都验证过。指挥官的判断力靠地面真事校准,完全不接触
事实就判不了结果真假;全自己干则上下文很快装不下,大项目不可能,
也没有并行。所以 Chair 必须既有感知工具又默认委派。

**观点 2:指挥的接触面不是代码,是两样东西——派出去的 prompt 和
收回来的结果评审。**
计划、契约、边界从 prompt 出去;对照契约找差异从评审回来。指挥的
肌肉长在这两处,不长在亲手改代码上。

**观点 3:合并的是认知,不是代码。**
代码的实际改动是 executor 干的活;Chair 收回来后做的是读 diff、
对照契约、对照方案、更新全盘认知。Chair 的产出物不是代码,是
"更新后的指挥状态 + 下一个动作"。Chair 亲手合代码就是滑回 DIY。

**观点 4:Chair 知道全盘地图,不是全盘代码。**
常驻脑中的:架构结构、模块契约、东西在哪、方案、目标、进度。
按需取阅的:具体代码。地图不是领土,靠感知工具随时放大局部。
把全盘代码装进脑子 = 上下文淹没的病根复发。

**观点 5:指挥能力不是模型内生的,是系统脚手架搭出来的。**
长程目标跟踪、权衡、知道何时介入——当前模型这些都弱。所以需要
外置载体、轮次、循环来搭脚手架。修指挥能力 = 修脚手架,不是换
更"聪明"的用法。

**观点 6:Chair 执行时,全场无人指挥。**
Chair 的时间是全场最稀缺资源。"这个活 Chair 自己会做"不是亲手
理由——亲手干的机会成本是整轮停滞。Chair 亲手只做四类:指挥
工作本身、感知、写不成契约的活、小到不值一张卡的活。

**观点 7:感知和执行要分开。**
工具给 Chair 是为了感知(侦察、抽查、验证、看关键文件),不是
为了执行。1.0 的 07-20 变更错不在"给了工具",在"给工具的目的
是执行",于是 AI 滑向全 DIY。感知常用,执行进卡片,分类写在
prompt 层,程序不门禁。

## 二、关于认知与记忆

**观点 8:Chair 无记忆,层即记忆。**
Chair 不是常驻进程,每次 Run 是"读事实→推进→写事实→结束"。
连续性不在进程里,在记录里。所以目标、信息、方案、推动点都必须
是外置载体,不能是 Chair 的脑内状态。

**观点 9:项目是被信息结构推进的,不是被执行动作推进的。**
事实层、信息层、目标与进度、当前方案、推动点、收敛口——每次
Chair Run 把这个栈往下推一格。项目动没动,看栈更新没有,不看
执行了几个任务。串行并行、效率,都是下游细节。

**观点 10:运行层连接过去和未来。**
已经做好的(代码现状、Progress、历次结果)是事实;未来想做的
(GOAL、方案)是意图;Chair 站在中间把新事实合进旧认知、再跟
方案比对、决定下一步。这就是指挥的全部内容。

**观点 11:程序管事实层,事实层以上全是判断领土。**
这是"程序只能记录"冻结规则的推论:程序越过事实层就是在指挥
领土上抢权。1.0 清理过的状态机、门禁、评分、自动验收,2.0 永不
回潮。

## 三、关于委派经济学(1.0 停摆的病根)

**观点 12:AI 一有工具就不指挥,不是能力缺陷,是成本结构问题。**
派一个 task 的成本:临时写完整契约(贵)+ 等待 + 子质量不可预测
+ 可能静默空跑。自干的成本:直接做,单次成功率高。Chair 每次都
做局部理性选择,累积成全局坏选择——跟 micromanaging 创始人一个
道理。**修法是降委派成本,不是收工具。**

**观点 13:契约要批量预写,不要派发时现写。**
执行卡片在轮次规划期批量写好,派发时 Chair 只翻牌。委派边际成本
≈零,Chair 才没有理由不派。

**观点 14:子任务必须可观察。**
长任务 checkpoint 到命名路径,Chair 随时看进度,不安就缩权、
重派、取消。指挥没有遥测 = 焦虑和不信任 = 更多 DIY(07-21 实证)。

**观点 15:loop 只在 DIY 不可能时才自然发生。**
任务规模不够,Chair 自己干得完,就不会派。这不是指挥模式错了,
是规模没到。所以 2.0 用轮次和卡片**结构性**地制造委派,而不是
指望 Chair 自觉。

## 四、关于并行、串行、线路

**观点 16:并行/串行不是执行技巧,是卡片的结构属性。**
两张卡无依赖 = 天然可并行;B 卡吃 A 卡产出 = 天然串行。派发时
Chair 机械翻牌,不每次重新想"并行还是串行"。功夫在写卡时:
该并行的别串起来(白等),该串行的别假装独立(双双返工)。

**观点 17:大并行是线路,小并行是线内阶段,串行主体是线内。**
一条线路 = 同一子目标 + 不相交写面 + 自含上下文。真并行只发生
在线路之间(写面不相交、上下文自含、各自可验收)。一轮 2~5 条
线,6 槽容量自然映射为 N 线 × 1~2 张在飞卡。

**观点 18:线路是故障的防火墙。**
一条线 stall,Chair 缩权重派那一条,其他线照跑。没有线路概念时,
一处卡死全轮阻塞(07-21 实证)。

**观点 19:并行是放大器,放大的是路线质量。**
指挥逻辑 × 串行 = 慢但正确;执行逻辑 × 并行 = 更快走错路(1.0
配方);指挥逻辑 × 并行 = 强大的指挥室(2.0 目标)。顺序必须
先指挥逻辑,后并行。1.0 恰好做反。

**观点 20:先划线,再排卡。**
平铺一堆卡再两两判依赖是 O(n²),必漏。依赖分析缩到线路级——
几条线一眼看清。线尾设合流卡:唯一允许多依赖的卡,合流点是
设计出来的串行点,不是意外撞出来的。

**观点 21:纵横之争落在线路数量和资源分配上。**
横向推进 = 多开覆盖线;纵向深挖 = 集中资源喂一两条线。指挥逻辑
四问(阶段/瓶颈/优先级/可逆性)的产出就是:本轮开哪几条线、
每条线吃什么资源。

## 五、关于轮次、循环、门禁、收敛

**观点 22:轮次是指挥的时间结构。**
没有轮次边界,整合、验收、换路无处安放,项目靠惯性滑行——
这是三周走偏的机制性根源。轮次给了它们位置。

**观点 23:两个循环各管一级。**
小循环(卡级):翻牌→执行→回收→对照判据→收敛/改卡/重派。
大循环(轮级):整合→目标切片→方案+线路+卡组→执行期→卡组
清空→验收→写回→下轮整合。第零循环是人:随时插入,压过一切。

**观点 24:门禁只设在"判断能改变走向"的地方。**
人门守方向(意图对齐、授权、体验),AI 门守事实(规划挑战、
收敛、核验),执行路径无门。门多了 Chair 变盖章机,回到 1.0。

**观点 25:收敛的锚是目标,不是方案。**
发散 ≠ 没按方案走;发散 = 事实表明方案不再服务目标。收敛动作
有两种:执行拉回方案(细节偏,改卡),方案拉回目标(方向偏,
提前收轮重整合)。1.0 只有第一种,于是方案错了也硬着头皮执行完。

**观点 26:换路条件要预先写。**
PLAN.md 里写"出现什么事实就换路"。发散检测是规划期写好的,
不是出事后现想的。

**观点 27:验收的两级。**
卡级收敛归 Chair(对照判据);轮级验收 Chair 呈报 + 人体验门。
验收未过补卡续轮,不设"失败"程序状态。

**观点 28:轮次大小 = 一次"人愿意体验一次"为限。**
太大 = 瀑布,失去整合的转向价值。

**观点 29:两种运行模式防官僚化。**
日常流(琐碎小事直接做或临时派单卡)+ 轮次(阶段级推进)。
不是所有事都开轮,否则系统被自己的流程压死。

## 六、关于人与 AI 的分工(2.0 最大的转向)

**观点 30:人从技术判断链里退出。**
非技术 Owner 判断不了技术、验收不了代码、写不出目标和边界。
方案必须按这个现实设,不能假装人能守住技术门。

**观点 31:人只剩三个角色:意图源、体验者、授权者。**
说想要什么(大白话,模糊没关系);试用后说感觉对不对(主人
判断,不是技术判断);批准不可逆动作(所有权,AI 永远不能替:
钱、公开、凭据、删除)。

**观点 32:认,比写,门槛低得多。**
人不用会写目标,只需认——"AI 说我理解你要的是 X"——点头或
摇头。意图对齐门按"认"设计,不按"审"设计。

**观点 33:锚下移一层:意图是锚,目标是诠释。**
GOAL.md 双层:意图层(人的原话,原样记录,AI 永不改写)+
目标层(AI 的结构化诠释,必须引用意图原话)。冲突时意图赢。
这样 AI 可以写目标,但空转在追溯时现形。

**观点 34:AI 自目标自验收是新风险,必须配四道药。**
AI 出题、AI 答题、AI 判卷 = 闭环自嗨风险。药:①现实锚(验收
挂客观事实:能跑的检查、能打开的页面、能演示的 demo,不对着
感觉验收);②反方制(目标层新建/大改时 opposition 必开);
③独立核验(高风险结果派另一上下文的 AI,只暴露差异);
④意图可追溯(任何目标指回人的原话)。

**观点 35:对人说结果、选项、后果,永不倒技术细节。**
"A 快但以后要返工,B 慢但稳,我建议 B"——人选的是偏好,
不是技术。非授权类问题带默认项("你不回,我按推荐走"),
技术问题绝不阻塞等人。轮末白话报告:给知情权,不给审批负担。
只在三种时刻问人:意图歧义、偏好岔路、授权。

**观点 36:第三个 AI 是体系本身。**
单次 AI = 一张卡;AI-AI = 完整自然语言 + 程序客观事实的回路;
AI-AI-AI = 体系化运转(轮次/线路/角色/循环/门禁),让千百次
AI-AI 交换随时间朝目标收敛。靠体系设计,不靠单次聪明。

## 七、1.0 诊断(观点的证据基础)

1. **Chair 没有地图**:全盘认知混在 prompt 和 Progress.md 里,没有
   独立载体,每次唤醒都不是清醒的脑袋。
2. **只有执行逻辑,没有指挥逻辑**:从没给过"为什么走这条路、
   什么情况下换路"。路线决策没人负责回答,Chair 只能凭当下感觉
   拍脑袋——不是能力缺陷,是从没被发给指挥的弹药。
3. **委派经济学坏掉**(观点 12~15):实证 07-20 开放直接执行后,
   07-21 全天 18+ 次 Run 零委派。引擎挂着挡,Chair 不踩油门。
4. **没有轮次边界**(观点 22)。
5. **人门设错了位置**(观点 30):默认人会审方案、验收结果、给
   目标——门形同虚设。

# 第二部分:体系设计(结论)

> 以下所有结论都是第一部分观点的推论。改结论,先改观点。

## 八、总原则(不变)

- **AI-AI-AI**:prompt 传完整信息,子 AI 输出完整自然结果;体系化
  运转靠设计不靠单次聪明。
- **程序只能记录**(冻结规则,永久):程序不判断、不门禁、不调度、
  不验收、不推进。
- **Chair 无记忆**:连续性是组织的,模型进程是一次性的。每次 Run
  从外置载体重建指挥状态。

一句话结构:**轮次管时间,线路管空间,卡片管动作,Chair 管判断,
程序管记录,人管意图、体验、授权。**

## 九、六层载体

| 层 | 载体 | 谁写 | 谁读 | 纪律 |
|---|---|---|---|---|
| ①事实 | 代码仓、`.deer-flow/` 记录、Progress.md | 程序记客观的;Chair/子写认知性事实 | 全体 | 只增不删 |
| ②信息 | STATE.md:全盘地图+当前认知+已知风险 | Chair(收敛时更新) | Chair 每次 Run 必读 | 地图不是领土,只放指针和判断;超限由 Chair 提炼归档 |
| ③目标 | GOAL.md(双层,见下) | 意图层=人;目标层=AI | Chair 必读 | 短 |
| ④方案 | PLAN.md:当前路线+理由+换路条件 | Chair | Chair 必读 | 旧案进 `plans/archive/` |
| ⑤推动点 | 卡组 DECK(按线路组织) | Chair 决策,程序记任务态 | Chair | 程序不调度不排序 |
| ⑥收敛口 | 卡级收敛 + 轮级验收(活动,非文档) | Chair | — | 产出落 STATE 与 Progress |

### GOAL.md 双层结构(观点 33)

```
意图层  人的原话,原样记录,AI 永不改写          ← 锚
目标层  AI 对意图的结构化诠释:目标、边界、价值优先级 ← AI 维护
```

- 每个目标层条目必须引用意图层原话(可追溯)。
- 意图与诠释冲突时意图赢;Chair 用大白话回来跟人对齐。
- 价值优先级示例:"跑通优先于健壮;横向覆盖优先于纵向深挖"——
  Chair 遇岔路查优先级,不猜。

### 卡片(执行单元,观点 13)

```
Card <轮次号.序号>: <标题>
├─ 方案节点    挂在 PLAN 哪个节点下(为什么存在)
├─ 目标        做什么,一段自然语言
├─ 边界        必须/禁止、涉及的精确路径
├─ 完成判据    可观察的"做完了长什么样",禁虚词
├─ 产出        命名输出路径
├─ 依赖        前置卡号(事实记录,程序不强制)
├─ 建议角色    executor / fact-finder / …
└─ 检查点      长任务的落盘节奏
```

粒度学说:一张卡 = 一个子 AI 一次能完成、结果能验证的最小独立单元。
太大→空跑风险;太小→派发成本吃掉并行收益。

### 线路(真并行单位,观点 17~21)

- 一条线路 = 同一子目标 + 一片不相交写面 + 一份自含上下文的一串卡。
- 一轮 2~5 条线。线内默认串行,阶段点允许小并行(须标注写面不相交)。
- 线尾设**合流卡**:唯一允许多依赖的卡,等所有汇入线到位。
- 线路随轮次生灭,下轮重划,不背上线包袱。

## 十、两个循环

**小循环(卡级,小时级)**:翻牌 → 子AI执行 → 回收 → 对照卡判据
→ 收敛 / 改卡 / 重派。

**大循环(轮级,天级)**:整合(事实+信息→STATE)→ 本轮目标切片
→ 方案+线路+卡组 → 执行期(无数小循环)→ 卡组清空 → 验收 → 写回
→ 下轮整合。

**第零循环(人)**:随时插入:新意图、改主意、叫停。Chair 停在
安全点、记现场、重规划。

## 十一、门禁(观点 24、30~35;程序永不设门)

| 门 | 守者 | 时机 | 检查什么 |
|---|---|---|---|
| 意图对齐门 | 人(只需认/否) | 目标层新建或大改 | AI 的诠释对不对 |
| 授权门 | 人 | 不可逆/钱/公开/凭据 | 永远归人,AI 不能替 |
| 体验门 | 人 | 轮末产出可体验物 | 试用感觉对不对 |
| 规划门 | opposition(AI) | 目标层新建/大改必开;其余 Chair 定 | 方案+卡组的反方挑战 |
| 收敛门 | Chair | 卡回收、合流卡、轮验收 | 结果对得上判据吗 |
| 核验门 | 独立 AI | 高风险/事实冲突/无法核实 | 只暴露差异,不裁决 |

执行路径无门:翻牌、传输、执行是机械动作。

## 十二、Chair

### 五步 Run 合约

1. **读**:GOAL / STATE / PLAN / 当前 DECK / INBOX(各有界)
2. **收敛**:INBOX 逐条对照卡判据 → 已收敛 / 差异待解(事实记录)
3. **翻牌/改卡**:依赖已满足的卡翻出;被新事实推翻的卡改或补;
   小卡直接做
4. **写回**:STATE / PLAN / DECK / Progress
5. **结束**:卡组空→轮验收;否则无状态结束,等唤醒

### Chair 亲手只做四类(观点 6)

指挥工作本身;感知;写不成契约的活;小到不值一张卡的活。

### 工具用法(观点 7)

感知工具直接常用;执行工具默认进卡片。分类写在 prompt 层,
程序不门禁。

## 十三、角色(一次性分身,按循环时刻上班)

| 时刻 | Chair | 角色 | 人 |
|---|---|---|---|
| 整合期 | 收拢事实、更新 STATE | fact-finder 补调查 | 认本轮目标 |
| 规划期 | 起草方案、划线、写卡组 | opposition 挑战(目标层大改必开) | 意图对齐门 |
| 执行期 | 翻牌、收敛、改卡、感知 | executor 跑写卡;fact-finder 跑读卡;auditor 在合流点/高风险点核对 | 随时插入 |
| 验收期 | 对照目标裁决、呈报 | recorder 落档 | 体验门 |
| 轮间 | 收轮或开新轮 | project-manager 提议下阶段 | 定夺(白话选项) |

名册:executor、fact-finder、opposition、recorder、project-manager
+ 五审计(runtime-reliability、persistence-migration、
frontend-protocol、security、platform-ops)+ general-purpose、bash。
8 个孤儿包(boundary、capability-governor、conflict-mapper、
debt-curator、evidence、freshness-keeper、learning-curator、
project-steward)落地前逐个裁决:映射、归档或删。

## 十四、委派经济学(观点 12~15)

- 契约在规划期批量预写(卡片),派发边际成本≈零。
- 长卡必须 checkpoint 到命名路径;Chair 感知抽查,不安则缩权/重派/取消。
- 子任务结果 = 自然语言全文 + 工件引用 + 程序附客观事实(改了哪些
  文件、退出码、时长、字节)。
- 同卡返工 2 次未果 → 升级人(判断,非程序门禁)。
- 子任务失败顺序:重写契约重派 → 拆小 → Chair 亲自 → 升级人。
- 6 槽是容量不是配额;依赖满足的卡全翻。

## 十五、收敛学说(观点 25~27;四层网)

1. **卡判据**:跑偏当场抓住。
2. **检查点**:长卡中途落盘,Chair 中途纠偏。
3. **换路条件**:PLAN.md 预先写"出现什么事实就换路"。
4. **轮末验收**:整轮漂移兜底。

细节偏→小循环改卡;方向偏→提前收轮重整合。不拿新卡去补错方向。
防空转四药见观点 34。

## 十六、Chair↔人 沟通协议(观点 35)

- 说结果、选项、后果,永不倒技术细节。
- 非授权类问题带默认项;技术问题不阻塞等人。
- 轮末白话报告:发生了什么、可试什么、决定了什么、下一步。
- 只在三种时刻问人:意图歧义、偏好岔路、授权。

## 十七、程序边界(允许清单,不多一件)

传输完整 prompt/结果;跑/杀一次性进程;硬超时与取消;进程清理;
子完成唤醒 Chair(唯一 loop 驱动);记录事件/audit/INBOX/客观 diff
事实;认证、凭据隔离、owner 隔离;呈现(UI 展示记录,不展示判断)。

**永不**:调度、排序、催办、门禁、验收、评分、状态机、自动返工。

## 十八、两种运行模式(观点 29)

- **日常流**:琐碎安全有界的小事——Chair 直接做或临时派单卡。
- **轮次**:阶段级推进——值得规划、验收、跟人对齐的事。

## 十九、1.0→2.0 迁移顺序

1. 文档先行:本蓝图审定 → GOAL.md 模板(意图层+目标层)、
   STATE.md/PLAN.md/卡片格式。

5. 不动:冻结规则、codex 传输、唤醒机制、工具组、channels、
   Progress.md 格式。
