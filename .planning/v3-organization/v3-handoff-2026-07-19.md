# NextOS V3 交接（2026-07-19）

## 1. 交接结论

NextOS V3 的组织运行基座已经完成，当前没有证据支持继续建设一个新的“第三切片”。下一步应转入一个边界清楚的真实项目试运行，观察 AI 组织是否按既定方法自然协作，再用真实失败证据决定是否改 Prompt、Role Skill、AGENTS 或产品能力。

本轮收口只保存当前 V3 实现、规划、验证与交接事实，不改变以下最高边界：

- Human Owner 提供兴趣、方向、非目标、现实权限与回人边界。
- Command Room/Chair 是唯一持续持有目标、上下文、计划、取舍与下一步决定的 AI。
- 程序只记录、传输、运行、取消、超时、隔离、保存客观事实和唤醒，不理解、评价、审批、排序或推进 AI 工作。
- 固定专业角色是可持续优化的 Prompt/Skill 上下文；子 AI 是一次性执行实例，返回完整自然结果后结束。

## 2. 已完成能力

1. **计划与指挥链**
   - 新的实质性执行方案或方向性修订：一个 Planner → 一个 Opposition → Chair 形成执行方案 → Human 自然语言讨论确认 → 执行。
   - 已确认方案内的执行、普通修复、停止低价值工作和有界优化由 Chair 直接继续，不重复走 Planner。
   - 阶段结果改变下一路线时：Project Manager 下一阶段建议 → Opposition → Chair 下一阶段方案 → Human 讨论确认。
   - 六条是单个 Chair 的并行容量，不是必须凑满的任务数。

2. **AI 组织与运行**
   - Goal Mandate、Operating Brief、Organization Map 使用同一追加式 Workspace 事实源。
   - Goal Cell 支持有边界的递归临时组织和完整结果回传。
   - 固定角色包已覆盖 Planner、Project Manager、Executor、Fact Finder、Opposition、Recorder，以及运行可靠性、持久化迁移、前端协议、安全和平台运维审计。
   - 每个 child handoff 要给出完整目标、事实、边界、完成标准，以及精确 working/input/output paths。

3. **资源与生命周期**
   - Gateway-local、内容无关 FIFO：全局运行 12、等待 64、单个 `(user, Command Room)` 最多 6 个 queued/running child。
   - 程序不根据角色、Prompt、结果质量或阶段选择调度顺序。
   - 已取消的 child 记录为 stopped；因 interrupted/cancelled 产生的 Chair wake 不重派，不再形成转圈。

4. **事实恢复与驾驶舱**
   - Workspace History 是 owner-scoped、原文、有限分页、只读的事实历史；不新增表、事件类型或语义状态。
   - Chair 只在需要旧事实时按需读取一页历史；每轮仍只自动注入最新 Mandate、Brief、Map。
   - Work Record 惰性加载 History，可查看已确认结果与投递事实，不提供审批、验收或推进控件。

## 3. 下一步工作优先级

### P0：真实项目试运行

在 `http://127.0.0.1:2026` 新开一个边界清楚、可停止、无生产写入的真实项目对话，观察完整链路：

1. 新实质方案是否只先派一个 Planner。
2. Planner 完整返回后，Opposition 是否收到原始任务与完整方案。
3. Chair 是否综合正反结果形成执行方案，并停下来与 Human 讨论。
4. Human 确认后，Chair 是否按真实独立工作域决定并行数量，而不是固定派 1 条或凑 6 条。
5. 每个 child 是否获得精确工作地址、输入地址与输出地址。
6. 阶段结果改变路线后，是否自然进入 Project Manager → Opposition → Chair 下一阶段方案 → Human 讨论。
7. 停止或取消后是否真正停止，不重派、不空转。
8. Chair 是否只在当前 Brief 不足时读取历史，而不是每轮加载全部历史。

试运行只记录实际摩擦、失败 Prompt、缺失事实和错误决策。不要为了“看起来完整”预先增加工作流、状态机、角色或 UI。

### P1：三项维护工作

1. **补一条带真实本地授权会话的 owner 全链路证据**：多 owner 隔离、多版 Workspace、真实 child result、acknowledgement、Chair History tool。不得为冒烟伪造认证或扩大数据权限。
2. **给新增五类专业审计角色补紧凑的正反向评测**：优先做一个共享的小型 role-skill probe pack，验证角色边界、证据分级与禁止越权；不要扩大 Skill 正文。
3. **把 AGENTS 指令链压回 32 KiB 内**：当前 backend 链约 `33,645` bytes，超约 `877` bytes；frontend 链约 `33,154` bytes，超约 `386` bytes。建议从仓库 `AGENTS.md` 无语义压缩至少约 1.2 KiB，把方法性说明移到本规划或 Skill reference。永久冻结的“程序只能记录”小节不得修改。

### 由真实摩擦触发，不预建

- 驾驶舱 queued/running/finished 容量概览：只有真实排障表明现有 TaskLane 不足时再做。
- 跨 revision 长文本/产物阅读体验：只有试运行证明检索困难时再做；不做自动摘要、评分或证据等级。
- 交互式组织图：只有 Map 原文与 Goal Tree 无法表达实际组织关系时再做。
- 项目 RAG：只有本地文件和 Workspace 历史已大到 Chair 无法有效检索时再讨论。
- 跨项目索引：先确认可见性、权限与保留边界。
- durable multi-Gateway queue：只有多实例或重启后必须继续 child 成为真实需求时再确认架构。

## 4. Skill 与 AGENTS 审查结论

### 当前不需要语义更新

- `skills/custom/nextos-commander/{AGENTS.md,SKILL.md}` 已覆盖最新确认的计划链、阶段链、并行容量、精确路径、按需历史、Goal Cell、acknowledgement 非验收和停止不重派。
- Commander SkillOpt 已覆盖方案暂停、六条非配额、阶段 handoff、结果恢复、精确路径与递归 Cell；本次 train/val/test 的 hard、soft 均为 `1.0`。
- 仓库、backend、frontend AGENTS 的 AI-AI 边界与当前实现一致，不应再复制更多同义规则。
- 性能选择“保留全部 MCP/plugins/Skills”是当前环境事实，留在 `Progress.md`，不应写成组织 Skill 或通用 AGENTS 规则。

### 需要的更新是维护，不是加规则

- AGENTS 只做无语义瘦身，解决指令链超预算；不要动永久冻结小节或改变 Chair/程序权限。
- 新专业审计角色已经有正确的窄角色包和静态映射测试，但还缺真实任务的正例/反例评测。下一步应补评测，不是继续扩写 Skill。
- 当前不新增角色。以后出现视频文案等反复使用的专业工作时，先用一次性 Prompt 验证；只有形成稳定 owner、输入、输出、禁止行为和评测后，才晋升为固定角色包。

## 5. 明确不要恢复的建设

- 工作流引擎、阶段状态机、程序审批/验收/评分、自动 Reviewer、自动返工、自动 Project Manager、自动角色选择、语义路由、自动 Skill 修改、图数据库和持久化消息队列。
- 已停止的宽泛审计不能自动成为 V3 第三切片。其 P2/P3 发现（超大 Goal Tree 截断、条件式 CORS、轮询成本、错误反馈、raw HTML 风险、运维硬化）作为独立技术债证据保留，只有 Human Owner 另行选择后才处理。
